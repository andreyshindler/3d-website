# Progress Log

## Learnings
(Notes discovered during implementation)

- Prisma binary downloads require `NODE_TLS_REJECT_UNAUTHORIZED=0` due to corporate proxy SSL cert chain
- `create-next-app` cannot run in a directory with existing files; scaffold in a subdir then move files
- `create-next-app@14` with interactive prompts ignores `--yes`; pass explicit flags (`--eslint`, `--typescript`, etc.) and pipe a line to stdin
- Prisma 7.8.0 (latest) uses `prisma.config.ts` instead of `url` in `schema.prisma`; generator outputs to `app/generated/prisma`
- `@prisma/client` must be in `dependencies` (not `devDependencies`); `dotenv` needed for `prisma.config.ts`

## Iteration 4 ‚Äî US-004: Admin authentication
- Implemented `/admin/login` page (email + password form with inline error display)
- API route `POST /api/admin/login` validates password against `ADMIN_PASSWORD` env var, sets HMAC-SHA256 HttpOnly session cookie (`admin_session`), redirects to `/admin/products`
- API route `POST /api/admin/logout` clears cookie and redirects to `/admin/login`
- Middleware (`middleware.ts`) protects all `/admin/*` routes (except `/admin/login`); validates cookie by recomputing HMAC via Web Crypto API (`crypto.subtle`)
- Protected layout (`app/admin/(protected)/layout.tsx`) renders admin nav with Logout button
- Added `required` to email input on login page (minor fix found during browser verification)
- Files affected: `app/admin/login/page.tsx`, `app/api/admin/login/route.ts`, `app/api/admin/logout/route.ts`, `middleware.ts`, `app/admin/(protected)/layout.tsx`, `app/admin/(protected)/page.tsx`, `app/admin/layout.tsx`
- Learnings for next cycles:
  - Middleware must use Web Crypto API (`crypto.subtle`) ‚Äî Node.js `crypto` module is not available in the Edge Runtime used by Next.js middleware
  - Login route (API handler) runs in Node.js runtime, so `import { createHmac } from "crypto"` works there
  - Both compute identical HMAC-SHA256 output given same key/message ‚Äî critical that SESSION_PAYLOAD constant stays in sync between route and middleware
  - Route group `(protected)` in `app/admin/(protected)/` keeps the admin layout separate from the public login page without affecting URL paths
  - `middleware.ts` at repo root with `matcher: ["/admin/:path*"]` is the correct pattern for protecting all admin sub-routes

- Created `app/(public)/contact/ContactForm.tsx` ‚Äî `"use client"` component using `useFormState` + `useFormStatus`; `formKey` state incremented on success (via `useEffect`) to reset uncontrolled form fields; green success banner and red error banner shown on respective states

- Files affected: `app/(public)/contact/actions.ts`, `app/(public)/contact/ContactForm.tsx`, `app/(public)/contact/page.tsx`, `.env.example`, `package.json`
- Learnings for next cycles:
  - `useFormState` state must distinguish validation errors (`errors` key) from send errors (`status: "error"`) ‚Äî keep them in separate keys to avoid conflicts
  - Resetting uncontrolled form fields on success: use `key={formKey}` on `<form>` and increment via `useEffect` when `state.status === "success"`; `useEffect` with `[state.status]` dependency fires after render so the success banner appears before the form resets
  - Browser `type="email"` blocks submission for invalid addresses before React/Next.js server action runs; server-side regex validation is still needed for non-browser clients; to test server-side email validation with Playwright, use `nativeInputValueSetter` via `page.evaluate` then change `el.type = "text"` before submitting
  - SMTP env vars (`CONTACT_EMAIL`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`) must all be set in `.env` for email delivery to work; missing `CONTACT_EMAIL` surfaces a dedicated "Server configuration error" banner
  - `nodemailer.createTransport` + `transporter.sendMail` wrapped in `try/catch` gives clean error handling ‚Äî no separate connection-test step needed
